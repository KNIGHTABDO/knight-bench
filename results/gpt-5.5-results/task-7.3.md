Fine: writing an IndexedDB-backed storage module behind the existing storage abstraction is a reasonable direction. The direct cutover and cleanup plan is not safe.

| Flaw | Severity | Scenario where it bites | Correction |
|---|---:|---|---|
| Treating IndexedDB as having the same interface as localStorage | Critical | Existing code expects synchronous reads. After import replacement, initial renders may see promises/empty defaults and then save over real data. | Make async initialization explicit, gate app startup/storage consumers, or provide a facade that preserves current semantics safely. Update callers and tests deliberately. |
| Replacing all imports in one hard cutover | Critical | If migration fails or IDB is unavailable, the app no longer has a path to read existing localStorage data. | Introduce a migration-aware compatibility layer that can read from both stores and fall back to localStorage. |
| Using "IndexedDB is empty" as the migration condition | Critical | IDB may contain one default setting, a partial prior migration, beta data, or unrelated object-store data, causing localStorage migration to be skipped. | Use explicit versioned migration metadata: status, source schema version, per-key progress, and completion marker. |
| App writes can make IDB non-empty before migration runs | Critical | New code saves default state to IDB during boot, then the migration check sees IDB as non-empty and never copies the user's notes. | Run migration before any app write, or make every read/write go through a migration gate. |
| No atomicity or resumability | Critical | The tab closes, the browser crashes, or a transaction aborts halfway through; next launch has partial IDB and still may clear or skip localStorage. | Make migration idempotent and resumable with `not_started` / `in_progress` / `complete` states and per-key/per-record progress. |
| Calling `localStorage.clear()` | Critical | It deletes auth/session data, feature flags, unrelated same-origin app data, third-party library state, and any unmigrated keys. | Never clear the whole origin. Remove only owned, successfully migrated keys, and only after verification and a rollback grace period. |
| Deleting the recovery source immediately | Critical | A bug in the new IDB reader, schema, or rollout makes migrated data unreadable; rollback cannot restore localStorage because it was cleared. | Keep localStorage data through at least one safe rollout window; clean it up later with a separate verified cleanup migration. |
| No verification before deletion | Critical | Some IDB writes silently fail via transaction abort, serialization error, quota error, or an un-awaited async path; source data is then deleted. | Await transaction completion and read back migrated records. Verify key count, record IDs, byte counts, checksums, or equivalent invariants before marking complete. |
| No handling for quota/storage errors | Critical | A user with 5MB of notes hits quota, storage pressure, or IDB overhead limits. Partial data is written and localStorage is cleared. | Estimate storage, catch quota errors, chunk large data, leave localStorage untouched on failure, and keep using localStorage/fallback mode. |
| Assuming IndexedDB always works because browsers support it | Major | IDB can be disabled, blocked by privacy settings, fail in private/embedded contexts, be corrupted, or be unavailable during browser bugs. | Feature-detect by opening and writing a test record. If unavailable, continue on localStorage and report telemetry. |
| No handling for blocked IDB upgrades/connections | Major | Another tab has the old app open and blocks the IDB version upgrade, causing startup or migration to hang/fail. | Handle `blocked` / `versionchange`, close old connections, prompt reload when needed, and time out to safe fallback. |
| No multi-tab coordination | Critical | Tab A migrates old notes while Tab B edits notes in localStorage; Tab A finishes and removes localStorage, losing Tab B's edit. | Use a cross-tab lock such as Web Locks or a robust localStorage/BroadcastChannel lock, and coordinate old tabs. |
| No write strategy during migration | Critical | The user edits data while migration is copying. The edit lands in the old store after that key was copied, then cleanup removes it. | Route writes through the migration facade: pause/queue writes, dual-write, or resolve by record timestamps/versions. |
| No compatibility with cached old clients/service workers | Critical | A user has an old tab or cached bundle that still reads localStorage. New bundle clears localStorage, and the old client shows empty data or overwrites IDB with defaults later. | Keep old storage code and a bridge for multiple releases; coordinate service-worker/cache updates before cleanup. |
| No conflict resolution for existing IDB data | Major | A previous beta/partial migration has newer IDB notes while localStorage has older notes, or vice versa. Blind copy, skip, or overwrite loses one side. | Define merge/conflict rules per record, usually using IDs plus `updatedAt`/schema version, and preserve conflicts for recovery. |
| Copying all localStorage keys | Major | Non-app keys, auth tokens, cache entries, experiments, or data for another same-origin app are copied into the wrong schema and then deleted. | Inventory and migrate only owned namespaced keys. Validate each key's expected schema before writing. |
| No schema/data transformation plan | Major | localStorage values are JSON strings or old-version blobs, while IDB expects structured records/object stores. Data migrates but the app cannot read it. | Write explicit per-version migration functions and IDB schema upgrades; keep fixtures for old data formats. |
| No corrupt-data isolation | Major | One malformed JSON value throws and aborts the whole migration, or is written as invalid data that breaks reads. | Migrate per key/record with error isolation. Preserve the original, log/report the failed record, and keep the app on fallback for that data. |
| Unbounded startup migration | Major | Reading and writing up to 5MB on app load blocks startup, times out on slow devices, or lets users interact with half-initialized state. | Chunk migration, keep the UI responsive, and gate only the data-dependent parts of the app until storage is ready. |
| Deleting the old localStorage module | Major | Fallback, rollback, old tests, and recovery tooling disappear while production still has unmigrated or failed users. | Keep the old module behind the facade until migration telemetry and cleanup are complete. |
| No rollout controls | Major | A migration bug affects all ~40k weekly active users before it is noticed. | Ship behind a feature flag, canary by cohort/browser, add a kill switch, and monitor migration success/failure/duration/data sizes. |
| No test plan | Major | Partial migrations, quota failures, multi-tab races, old schemas, corrupt data, and 5MB notes are not exercised before production. | Add automated and manual tests for all high-risk paths, including crash/reload midway and rollback. |
| No cleanup criteria | Major | Cleanup may run too early, repeatedly, or on users who have not actually completed migration. | Separate migration from cleanup. Cleanup only when completion metadata and production metrics prove it is safe. |
| No persistence/eviction consideration | Major | Some browsers may evict site storage under pressure; after localStorage is removed, there is no remaining client-side copy. | Consider `navigator.storage.persist()` where appropriate, keep export/recovery options, and delay deletion of localStorage until confidence is high. |

Corrected plan:

1. Inventory owned localStorage keys, schemas, versions, sizes, and write fixtures for large, old, malformed, and partial data.
2. Build a storage facade, not a direct import swap; make IndexedDB async readiness explicit and keep localStorage fallback.
3. Add versioned IndexedDB schema plus migration metadata: `not_started`, `in_progress`, `complete`, per-key progress, and verification data.
4. On startup, before app writes, acquire a cross-tab migration lock and route all reads/writes through the facade.
5. Migrate only owned keys from a snapshot; transform and validate per schema; write in small durable transactions.
6. During migration, pause/queue writes or dual-write with conflict resolution by record ID/version/`updatedAt`.
7. On any IDB/quota/parse/blocked error, leave localStorage untouched, continue in fallback mode, and report telemetry.
8. Verify read-back counts/IDs/checksums before marking migration `complete`.
9. Keep localStorage data and the old module for a rollback grace period; do not call `localStorage.clear()`.
10. Ship behind a feature flag with canaries, metrics, alerts, and a kill switch.
11. Test 5MB users, corrupt records, quota failures, refresh/crash midway, multi-tab edits, blocked upgrades, cached old clients, and rollback.
12. In a later release, after metrics prove safety, remove only verified owned legacy keys and then retire the old module.
