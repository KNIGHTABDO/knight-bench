SSR renders a route on each request, so the server is hit when the user navigates or refreshes; they wait for HTML generated with current data, then hydration makes it interactive. SSG renders at build time, so normal visits hit static storage/CDN, not your server; users get very fast HTML, but content only changes after a new build. ISR is SSG plus timed or on-demand revalidation: visits usually get cached static HTML, while the server regenerates in the background or on trigger; users see fast pages that may be slightly stale. Streaming SSR still hits the server per request, but React sends HTML in chunks as data and components become ready; users see shell and finished parts earlier instead of waiting for the whole page. Choose SSG/ISR for cacheable pages, SSR/streaming for per-request data.
